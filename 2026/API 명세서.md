# API 명세서

## API 명세서란?
API를 이해하고 사용할 수 있도록 작성된 문서이다. API의 기능, 엔드포인트, 요청과 응답 형식, 인증 방식, 오류 처리 방법 등을 담고 있다. 쉽게 말해, API를 사용하기 위한 모든 정보를 깔끔하게 정리한 청사진 같은 것이다.

## API 명세서에 관해
GET, POST, PUT, PATCH, DELETE가 있다.

`GET` : 데이터를 받아온다.
`POST` : 데이터를 전송한다.
`PUT` : 입력한 데이터를 새로 추가한다. 기존에 이미 데이터가 있다면 덮어씌운다.
`PATCH` : 기존에 입력한 데이터를 수정한다.
`DELETE` : 데이터를 삭제한다.

## Request와 Response란?
`Request`는 서버에 보내는 요청
`Response`는 서버에서 보내는 요청

항상 Request가 선행되어야 이루어지는 작업이며,
Request가 없으면 서버 통신은 이루어지지 않는다.

### Request
```
/api/users?page=2
```
이 부분은 전송해야 하는 부분이다. 위는 GET일 때의 Request이고,

### Request
```
/api/users

{
    "name": "jaehyuk"
    "job": "student"
}
```
이는 POST일 때의 Request이다.

## 차이점
이렇게 `{}` 에 묶인 형태를 **BODY**라 부르고, `/api/users`는 **PATH**라고 부른다.

**BODY**는 말 그대로 보내는 몸체, 즉 덩어리를 뜻한다.
**PATH**는 길, 즉 어떤 경로로 이동해야 하는지를 뜻한다.

그런데 위의 **PATH**에서 `?page=2`라는 희한한 녀석이 있다.
이 녀석은 `QUERY` 라고 부르는 친구로, user의 목록 중에서도 특히 2페이지에 있는 녀석을 보고싶을 때 사용한다.

## PATH와 QUERY는 어떻게 차이가 있나요?
`PATH`는 주로 그 경로에 있는 데이터 그 자체를 식별하고 싶을 때 사용한다.
`QUERY`는 주로 정렬된 데이터를 구별해서 가져오고 싶을 때 사용한다.

## HEADER란?
이는 서버와 통신을 주고받을 때 포함된 `보조 데이터`를 말한다.
여기에는 가볍지만 꼭 보내줘야 하는 데이터가 포함된다,
앞서 나온 유저의 고유번호를 이로 전달할 수 있다.

하지만 단순하게 고유번호를 주고받는다면, 보안 등에서의 안정성 문제가 발생할 수 있다. 이를 위해서 나온 것이 바로 `TOKEN`이다. 로그인할 때 서버로부터 인증 토큰을 받고, 우리는 고정된 정보가 아닌 이렇게 인증된 입장권을 가지고 사용할 수 있는 것이다.
